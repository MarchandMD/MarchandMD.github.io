> I can’t imagine (though perhaps I’m not very imaginative) that Prompt, Context, or Specification “Engineering” would lead to a bright and prosperous profession for programmers.

This definitely seems to be the crux of the identity crisis of the title of this article.

> For decades, programmers have been pampered within companies. These narratives offer a new way for management to tip the balance back in their favor.

I haven't been a software engineer for decades, but I have had this thought: All this AI hype feels like it's a way for non-technical to target the technical and wrestle away the necessity of software engineers. It's like class warfare on the jobsite.

> FORTRAN didn’t try to eliminate a craft but built on top of it.

Is the point of AI to eliminate software engineers? Given the language of big tech companies, I'd say so....

> We prefer predictability, compositionality, idempotence, and integration tests that aren’t flaky. LLM code represents the opposite of that: inconsistent chaos.

Yeah, coding and software programs require precision and logic, however buggy it may be. Natural language is not, by it's nature, logical.

There are all these nuances of natural language like sub-text, innuendo, implicit meaning, shared understanding, sarcasm, etc. These concepts do not translate well to prompts or chatbot interactions.

>  I find that I don’t read the code an LLM generates for me as closely as I would if I had written it myself or had reviewed it in a PR.

This is why efficiency gains aren't necessarily objectively better, IMHO.

Reading a line of code is not the same as understanding a line of code that's been read.

Working in Ruby, the language _can be_ self documenting, if done right.

This should improve comprehension.

But generated code is so bountiful, it's a sizable effort to consume/comprehend, interpret with 100% accuracy, so it's a natural bottleneck.

> Taking the plunge into the abyss of one’s ignorance to reveal, learn, and understand a topic and its implications is both gratifying and crucial to good software.

This is still possible with AI. In fact, it's more immediate, and less risky.

I can continue to prompt for clarification of a topic/word and get really granular. I can ask what could be perceived as really dumb questions, to refine my own mental model.

Of course I'd have to have the tolerance for asking these things, and copping to the idea that I genuinely don't understand the piece of syntax I'm focused on.

And this is a vital skill. Pushing back and not trusting anything from the AI except for what I already know to be true _feels_ like a knowledge gain. I'm using the AI to actually learn a topic. Despite the topic or skill being very niche.

But isn't that what makes a really good SWE?

> ... and, often, away from the keyboard.

When was the last time a solution to a coding problem arrived the morning after an evening away from the work station? The idea of mulling over logic has evaporated seemingly instantaneously.

This makes me wonder: Where has that experience been relocated to?

>  it’s only when we write repulsive and repetitive code that we realize that there is a better, more succinct, elegant, compositional, and reusable way.

It is possible to refactor using AI into well-established design patterns.

I'm learning about design patterns now, and I'm beginning to be able to identify, when reading other peoples code, or messy diffs, what constitutes a code smell.

This is then revealing an opportunity to apply a design pattern. Did I arrive at the design pattern organically? I don't know if I ever would have. Being armed with the knowledge of what a design pattern is gives me the opportunity to be on the lookout for places to implement it.

> Rather than addressing the actual problems, we reached for something far more complex and nebulous to cure the symptoms.

Good habits and strong fundamentals for writing durable software seem to cure the cause. Taking the time to learn those shift the narrative of AI to being all-suggesting and not all-knowing.

> But I profoundly do not want to be merely an operator or code reviewer:

Reading code can be very insightful. It's the idea of active reading. And in codebases, and specifically PRs, actively reading is very, very important.

> I want to drive, immerse myself in craft, play in the orchestra, and solve complex puzzles. I want to remain a programmer, a craftsperson.

I'm thinking about a musical savant. Being able to listen to a piece of music and identify the key, the tempo, the structure of the song, upon a sample of the hearing of it is a completely legitimate and underrated skill.

Maybe this is where code reviewing will drive engineers?

There is no shame in being able to listen to a song, and identify the key parts of it, and then duplicate that song instantly. In fact, that's a super awesome flex for any musician with a well-trained ear.

## Conclusion

I do still side with the author on this: There is an element of software engineering that is being lost.

At the same time though, perhaps the requirements of working _with_ (not against) this new tool could allow for a different kind of progress and refinement of all the skills and attribute that make a software engineer a software engineer.
